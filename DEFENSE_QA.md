# ClassSense 答辩问答集

> 面向作品答辩，按"整体定位 → 技术实现 → 创新与边界 → 落地与未来"顺序组织。评委常问问题均给出标准回复范式，建议结合 Demo 使用。

---

## 一、项目定位与总体设计

### Q1. 这个项目解决的是什么真实问题？为什么值得做？

**A.** 传统课堂中，教师要么通过"点名提问""课后反馈"等滞后手段判断学生听课状态，要么依赖经验观察前排几位学生，难以覆盖 40+ 人的全班。ClassSense 让教师拥有一个始终在线的"第二双眼睛"：实时把全班专注度量化成分数与行为分布，课后一键查看时间线报告，找到讲课节奏上的薄弱点。它不是要取代教师，而是把课堂感知这件事 **数据化、可追溯化**。

### Q2. 系统整体架构如何，各模块分工？

**A.** 三层解耦：
1. **感知层**：摄像头 → YOLOv8-Pose 关键点提取 → 行为规则引擎判定五类行为 → IoU 轨迹跟踪做时间平滑；
2. **服务层**：FastAPI 异步后端，REST 负责课堂生命周期（上课/下课/历史）、WebSocket 负责实时推送聚合指标、MJPEG 负责调试预览流；
3. **应用层**：教师仪表盘（Chart.js 可视化）+ 课后报告（matplotlib）+ LLM 智能助教（周期性分析）。

关键一条：**AI 线程与 Web 事件循环完全解耦**，通过 `run_coroutine_threadsafe` 桥接，保证 UI 永远不卡顿。

### Q3. 为什么选 YOLOv8-Pose 而不是自己训练模型？

**A.** 三个理由：
1. 课堂行为的判别特征（低头角度、躯干压缩比、手腕高度）在 **骨骼空间** 比在像素空间稳定得多，所以只需要通用的 Pose 模型，不需要自定义分类器；
2. 自己采集课堂数据 + 标注成本极高，且容易过拟合到特定教室；
3. YOLOv8-Pose 是 Ultralytics 的 SOTA 之一，COCO 预训练权重对大部分姿态已足够鲁棒，**零训练即可部署**。

## 二、技术实现细节

### Q4. 行为是怎么判定的？为什么是规则引擎而不是神经网络？

**A.** 规则引擎基于 17 个 COCO 关键点的几何关系，判定顺序 **趴桌 > 举手 > 低头 > 扭头 > 专注**。举例：
- **趴桌**：躯干纵向比 = `|hip_y - shoulder_mid_y| / shoulder_width`，小于 0.3 + 鼻子几乎平肩 → 判定；或者脸部全遮挡 + 双肩可见。
- **低头**：`(nose_y - shoulder_mid_y) / shoulder_width > 0.25`。
- **扭头**：鼻子相对两耳中点的水平偏移超过 30%。

所有阈值都经 **肩宽归一化**，与人离摄像头远近无关。选择规则引擎而非 CNN 的核心理由是：
1. 可解释：任何一次判定都能倒推出关键点坐标，便于 debug 和阈值调整；
2. 零数据门槛：不需要行为标注数据集；
3. 速度快：纯 NumPy，开销忽略不计。

### Q5. 为什么不把多帧送进一个时序网络？

**A.** 时序网络（如 ST-GCN）确实更强，但本项目中被两个因素否决：
1. 目标学生的行为变化是**低频**的（秒级），单帧判定 + 滑窗投票已经能达到稳态；
2. 课堂远景下每人可能只占 60×120 像素，时序特征的增益边际递减，但算力代价是指数增长。

### Q6. 多人跟踪用的是什么？有没有用 DeepSORT / ByteTrack？

**A.** 我们用的是 **自研轻量 IoU 匹配器**（`person_tracker.py`）：
- 前后帧框按 IoU 降序做贪心匹配，保留 `id`；
- 每个轨迹维护一个窗口长度可配置的行为历史 deque；
- 需要 `≥ ceil(window/2)` 的多数票才切换到新行为，避免单帧抖动；
- 连续丢失 `max_missed` 帧才销毁轨迹。

没用 DeepSORT 的原因：Deep 特征要额外跑一个 embedding 网络，在 CPU 上拖慢 30%+；课堂场景学生基本不会跨位互换，IoU 足够。

### Q7. 怎么保证实时性？推理帧率大概多少？

**A.** 分层优化：
- **模型选型自适应**：默认 `yolov8m-pose`（mAP 67），弱机器可切 s/n；
- **设备自动检测**：有 CUDA 用 GPU + FP16 半精度（速度翻倍），无 GPU 回退 CPU 且不启用 half；
- **分辨率调平衡**：`imgsz=832` 是 CPU 下的"精度/速度"最佳点（1280→832 速度 ×2.3 精度几乎无损）；
- **模型预热**：初始化时跑一次 dummy 输入，避免首帧 JIT 编译拖时间。

实测：CPU i5-12 代约 3-5 FPS，GTX 1660 约 20+ FPS。**检测帧率与预览帧率解耦**（见 Q8），用户观感始终流畅。

### Q8. 网页画面为什么比 AI 检测跑得快？

**A.** 关键设计：**MJPEG 单向流 + 检测结果快照缓存**。
- AI 线程每产出一次检测，把"人框+行为"写入 `_stream_last_detection` 快照缓存；
- `/api/debug/stream` 以 30 FPS 目标从摄像头读取**最新原始帧**，叠加最近一次的快照数据渲染后 yield；
- 浏览器用 `<img src="/api/debug/stream">`，原生解码 `multipart/x-mixed-replace`，零 JS 轮询开销。

结果：画面接近摄像头原生帧率，框跟随略有滞后（检测 FPS 决定），但整体观感流畅。

### Q9. 为什么专注度平滑用"时间窗"而不是"帧数窗"？

**A.** 最初写的是 `maxlen = 5秒 × DETECTION_FPS`，相当于假定检测吞吐恒定。实际情况：CPU/GPU、分辨率、场景复杂度都会影响吞吐，经常出现"低 FPS 下平滑窗实际覆盖 20+ 秒"的问题。改为 **按时间戳裁剪 deque**，严格保证任何机器上滑窗都是 5 秒，让平滑行为与硬件无关。

### Q10. 数据库存了什么？为什么每 3 秒存一次？

**A.** 两张表：
- `ClassSession`：课堂元数据（id、名称、开始/结束时间、平均分）；
- `AttentionSnapshot`：3 秒粒度的时间线快照（秒数、总人数、专注度、行为计数 JSON）。

**不存图像**，只存聚合指标。3 秒是经验值：
- 更密（1 秒）会让 45 分钟的课产生 2700 条记录，报告图表密集难读；
- 更疏（10 秒）会丢失像"突然爆笑"这种短暂群体反应；
- 3 秒既能平滑偶发噪声，又能保留行为变化的细粒度。

### Q11. LLM 助教怎么工作？用的什么模型？

**A.** 采用 **OpenAI 协议兼容客户端**，用户可在 `llm_config.json` 填入任意兼容端点（官方 / 文心一言兼容接口 / 自建 vLLM 等）。助教管理器 `AgentManager` 在课堂启动时同步开始：
1. 以固定周期（比如每 60 秒）聚合近期的行为计数和分数变化；
2. 构造简洁的 prompt，调用 LLM；
3. 把回答作为"AI 洞察"推回前端显示在仪表盘右侧卡片。

LLM 失败不会影响核心流程——助教是增强，不是依赖。

### Q12. 前端为什么不用 React/Vue？

**A.** 因为没必要。页面只有三张：仪表盘、设置、报告；交互靠 WebSocket 推送 + Chart.js 重绘。用原生 JS 能避免几 MB 的 bundle、零构建流程，部署时直接 `pip install + python run.py` 全栈就位——这是评委能在自己电脑上跑起来的关键。

## 三、创新与边界

### Q13. 这个项目的核心创新点是什么？

**A.** 四点，按重要性排序：
1. **可解释的课堂行为规则引擎**：所有判定都能通过几何关系倒推，教师/研究者可以按教室布置调参；
2. **检测-预览解耦的流式架构**：AI 慢 vs 画面顺的经典矛盾通过 MJPEG + 快照缓存优雅解决；
3. **时间窗平滑 + IoU 贪心跟踪**：轻量、无训练数据、在低算力机器上仍稳定；
4. **数据-可视化-LLM 分析三位一体**：不是简单地"给教师看数字"，而是把原始行为流 → 量化指标 → 自然语言洞察打通。

### Q14. 和已有商业产品（如课堂行为分析 SaaS）有什么差异？

**A.** 三条差异：
1. **本地推理**：商业产品通常要把画面上传云端（隐私+带宽隐患），我们全程本地；
2. **开箱即用**：无需训练、无需采集，一份预训练权重直接跑；
3. **研究友好**：规则引擎 + 纯 Python，研究者可以替换任何一个模块做对比实验。

### Q15. 系统目前的局限？

**A.** 客观列出：
- **远景小目标**：后排学生像素占比过低时，姿态关键点置信度下降，会被过滤；
- **光照恶劣**：教室夜课只开单侧灯时，一半学生可能漏检；
- **遮挡**：被前排同学完全挡住的学生检测不到；
- **行为粒度**：目前只有五类，像"抠手机""写字"这类细行为规则引擎区分不了；
- **LLM 幻觉**：助教偶尔给出过度解读，我们通过限制 prompt 聚焦"可观测数据"来缓解。

### Q16. 会不会引发"课堂监控"的伦理争议？

**A.** 我们从三处做了边界设计：
1. **不存图像**：数据库只存聚合指标，无法用来"追踪"具体学生；
2. **不做人脸识别**：YOLOv8-Pose 只输出骨架，不识别身份；
3. **教师端而非管理端**：数据在上课老师本机，不聚合到学校服务器。
系统定位是**教学反馈**，而不是**学生评分**，这在产品层面就应当被明确写入使用规范。

## 四之补、隐私与合规专项问答

> 隐私是课堂类 AI 项目最容易被评委深挖的方向，这里按"信息收集 / 存储传输 / 身份识别 / 合规与知情 / 风险兜底"五个维度系统回答。

### Q16.1 系统到底采集了哪些数据？请逐项说明。

**A.** 精确列表：

| 数据类别 | 是否采集 | 是否落盘 | 备注 |
|---|---|---|---|
| 课堂视频流 | 从摄像头读取 | **否** | 仅在内存中处理，用完即扔，不写磁盘 |
| 原始帧图像 | 仅调试流启用时在内存渲染 | **否** | MJPEG 推给浏览器后立即释放 |
| 人脸图像 / 人脸特征 | **否** | **否** | 全流程没有人脸检测模块 |
| 17 点骨架坐标 | 在内存中用于判定 | **否** | 每帧判定完即丢 |
| 学生身份 / 学号 / 姓名 | **否** | **否** | 系统不知道任何学生个人信息 |
| 行为聚合指标（5 类计数、班级专注度分数） | 是 | 是 | 每 3 秒一条，存 SQLite |
| 课堂元数据（名称、起止时间） | 是 | 是 | 教师自己填 |
| 声音 / 麦克风 | **否** | **否** | 目前版本不用声学信号 |

一句话：**进入硬盘的只有"这节课有 X 人 + 每 3 秒 5 类行为计数 + 一个整体分数"，任何一条记录都无法定位到具体学生**。

### Q16.2 那调试预览里不是能看到每个学生的脸吗？算不算采集？

**A.** 这是一个好问题，也是我们专门设计的细节：
- 调试流是 **"渲染后推送、浏览器显示完即 GC"** 的实时管道，不在后端落盘也不在前端缓存；
- 这个开关**默认可关**（设置页"调试画面"开关），教师上课前一键关闭后，画面连内存都不留；
- 流只绑定在本地 `127.0.0.1`，不暴露到外网；生产部署时可通过反代加 IP 白名单/鉴权；
- 调试流中的人像是**实时摄像头原画叠加骨架框**，不做人脸处理——相当于让教师看到"系统看到的东西"，用途是让教师信任判定结果。

### Q16.3 摄像头本身的隐私风险怎么处理？

**A.** 三层约束：
1. **物理层**：摄像头对准学生的视角应当 **广角 + 偏高** 以看全班，而非特写某个学生；我们在部署手册里会写明"对准前排课桌以上，不得贴脸拍摄"；
2. **协议层**：`cv2.VideoCapture` 走的是本地 USB/UVC，不经过网络；
3. **开关层**：下课即调用 `stop()` 释放摄像头句柄，操作系统层面会关闭指示灯，学生能直接看到"没在录"。

### Q16.4 如果学校问"数据会不会被学校/第三方拿走"怎么答？

**A.** 标准回答：
- **技术上**：数据库是本机 SQLite 文件，教师端一台机器，不存在向云端/学校服务器汇总的逻辑；
- **架构上**：我们**没有做用户账号体系和数据同步**，这是有意设计，就是为了让项目天然不具备"学校级数据池"的能力；
- **合规上**：如果未来学校要做全校统计，也必须先有**教师知情同意**和**学生代表听证**的流程，这是产品边界而不是技术问题。

### Q16.5 未成年学生（中学/小学场景）有特殊要求吗？

**A.** 有，而且必须严格得多。我们把这一条写进了使用边界：
- **未成年场景需要监护人与学校双重知情同意**（参考《个人信息保护法》第 31 条、《未成年人保护法》第 72 条）；
- 这种场景下**建议进一步降级**：推理分辨率可降到 416，只输出班级总分、不展示个人框；
- 调试画面在未成年场景下**建议彻底禁用**（代码里已经可以通过 `runtime_config` 关）。
我们的立场是：技术允许不等于应该用，未成年场景要比成年严一档。

### Q16.6 有没有学生"反对被分析"的选择权？

**A.** 我们从两个层面支持：
- **系统层**：教师可以关闭调试流、可以随时下课、可以删除数据库文件——控制权完全在教师；
- **产品层**：建议开学第一节课做 1 分钟说明，告知学生"系统统计的是班级整体专注度，不识别个人"，学生若有异议可反馈给任课教师；
- **数据删除**：任何时候删除 `classroom.db` 就能抹掉全部历史数据，不存在"删不掉的云端副本"。

### Q16.7 LLM 模块会不会把课堂数据发到第三方？

**A.** 这是隐私设计里最微妙的一环：
- **默认关闭**：`llm_config.json` 里 `enabled: false`，不配置 API Key 不会发任何请求；
- **传输内容**：开启后发给 LLM 的只是**聚合后的文字描述**（如"过去 60 秒 5 人低头 2 人举手班级专注度 62"），**没有图像、没有骨架坐标、没有学生身份**；
- **用户须知**：LLM 端点由教师/学校自己配，我们建议优先用本地 vLLM 或私有化部署的模型，避免学生数据进入公开 API；这一条也写在 README 和设置页提示里；
- **可审计**：所有 LLM 调用都可以在代码里找到唯一入口（`app/agents/manager.py`），便于审计。

### Q16.8 万一系统被攻击，能泄漏出什么？

**A.** 威胁建模：
- **最坏情况**：攻击者拿到教师电脑的 SQLite 文件；
- **泄漏内容**：几次课的"班级专注度曲线 + 行为分布"，**不包含任何学生身份、画面、音频、骨架**；
- **危害等级**：远低于学校教务系统泄漏（那里有成绩/身份证/家庭住址）；
- **缓解措施**：Windows 下可以启用 BitLocker 给磁盘加密，进一步降低风险。

### Q16.9 为什么不做人脸识别？——做了不是能给个人反馈吗？

**A.** 这是一个**主动的设计决策**，不是"技术做不到"。拒绝人脸识别的三个原因：
1. **法律风险**：人脸是《个人信息保护法》明确的敏感个人信息，采集处理有更严苛的合规要求（单独同意、DPIA、存储加密等），学校场景的推广会被合规卡住；
2. **偏差放大**：如果把"专注度"挂在学生名字上，教师可能无意识地把 AI 分数等同于学生优劣，造成以算代人、标签化的负面效应；
3. **技术脆弱性**：课堂远景下的人脸识别准确率远低于骨架判别，给出"张三低头"很容易错成张三旁边的李四，风险高收益低。
**我们有意识地把这扇门关上**，而不是"以后再加"。

### Q16.10 数据留多久？会不会无限累积？

**A.** 目前实现里数据不会自动清理，但这容易补。答辩时可以提两层方案：
- **短期**：仓库里加一条 `retention_days` 配置，启动时清理超期快照；
- **长期**：只保留每节课的"平均分 + 行为分布快照"这一汇总视图，原始 3 秒级数据过 N 天自动丢弃，把最敏感的颗粒度降下来。
这是一个需要学校侧一起定义保留期的合规问题，我们只在技术上留了钩子。

### Q16.11 系统有没有做"差分隐私"或"联邦学习"这样的前沿隐私技术？

**A.** 坦白地讲，**没有直接用**。原因是：
- 差分隐私适合"大规模数据发布"场景，本系统的数据只留在单机，边界已经足够；
- 联邦学习适合"跨机构训练"场景，而我们用的是预训练模型，零训练数据，不需要联邦。
如果未来做跨校教研的数据共享（Q19 里提到的"多教室对比"），**那时才会引入差分隐私**——提前引入只会增加复杂度却不产生隐私收益。这是我们对"隐私技术应当按需引入"的原则性判断。

### Q16.12 一句话总结隐私立场？

**A.** "**边缘推理 + 不识别身份 + 不存图像 + 开关在教师手里**"。任何一项被打破，都等于违反我们的产品边界，这在技术上和产品上都要守住。

### Q16.13 为什么不给学生一个"退出采集"开关？看起来更尊重个人自主权。

**A.** 这是经过深思后**主动放弃**的设计，不是没想到——它在教育场景下会**放大而不是降低伤害**：

1. **可见的缺失 = 新的歧视信号**。退出者在采集信号里呈现为"空洞"，老师在仪表盘上能**直接看出谁退出了**。哪怕老师主观不想针对，缺失本身就会被解读为"这个孩子不合作"。
2. **霸凌风险**。同学之间会产生"老实采集派 vs 不老实派"的对立；家长群会攀比"他家孩子退出了是不是有什么问题"。
3. **维护成本错位**。真正想保护的是"不想被观察的未成年人"，结果开关却需要他们**主动承担社交代价**才能行使——这本身违背"对弱势方友好"的原则。

所以我们把退出权**向上集中到整个班级**：老师一键打开"隐私模式" → 全员马赛克 + 全员不落盘 + 全员不调 LLM。要退就全退，没有"谁退了谁没退"的差别化信号。这叫**Chilling Effect 规避**，是比盲目给 opt-out 更成熟的做法。

真正的家长/学生反对诉求，走学校行政流程：学校层面禁用该班级整套系统，而不是在运行时由学生个体承受社交代价。

### Q16.14 隐私模式具体会发生什么？

**A.** 一个开关控制三件事，全班级一刀切：
1. **调试预览马赛克**：人体框内强制像素化，无从辨认身份
2. **数据库零写入**：不再产生任何 `AttentionSnapshot` 记录
3. **LLM 全禁用**：`is_configured()` 直接返回 `False`，既不投喂也不调用

实时专注度曲线仍会显示——因为那是纯聚合数字（"78% 专注"），不含个人可识别信息，反而有助于公开课展示系统能力。

### Q16.15 API Key 安全怎么保证？

**A.** `llm_config.json` 中的 `api_key` 字段**自动加密落盘**：
- 机器密钥存于 `.secret_key`（权限 600，已加入 `.gitignore`）
- 加密方案：HMAC-SHA256 派生流密钥 + XOR + HMAC 完整性标签（纯标准库，零新增依赖）
- 密文统一前缀 `enc:v1:`，便于未来换算法
- **首次运行自动迁移**：如果历史 `llm_config.json` 仍是明文，启动时自动加密回写

这个方案**不声称**抵御具备本机文件读取权限的高级攻击者（那需要 TPM/HSM 级别硬件信任根），但能防住"截图泄露""误提交 GitHub""保姆级演示场景别人瞄一眼"这三类主要威胁。

### Q16.16 聚合统计的差分隐私是怎么用的？

**A.** `app/privacy.py` 提供两个基础原语：
- `k_anonymize(count, value)`：分组样本量 < 5 时返回 `*`，直接不发布
- `dp_release(value, ε=1.0)`：Laplace 机制加噪后裁剪到合法区间

合并版 `safe_group_stat()` 在当前项目尚未跨班级/跨年级时**作为接口预留**。一旦扩展到"全年级专注度对比"这类场景，所有聚合输出**必须**过一遍它，否则不允许上线。这是架构层面的"隐私默认开启 (privacy by default)"。

## 四、落地与未来

### Q17. 部署难度？学校要准备什么？

**A.** 极低：
- 硬件：一台带摄像头的电脑（笔记本/讲台机都行），推荐带 GTX 1050+；
- 软件：Python 3.10+ + `pip install -r requirements.txt`；
- 权重已经随仓库上传，开箱即用；
- 摄像头装在教室后方对准讲台前的学生即可。

### Q18. 性能还能怎么提升？

**A.** 三条路径：
1. **ONNX Runtime / TensorRT 导出**：CPU 上 ONNX 比原生 PyTorch 快 1.5×；GPU 上 TensorRT 能到 3×；
2. **半身模型**：训练一个只预测上半身 9 个关键点的定制模型，推理能快一倍；
3. **跳帧预测**：把相邻两帧的人体框做线性外推，检测 FPS 可进一步降低而视觉上不掉帧。

### Q19. 未来会加哪些功能？

**A.**
- **声学信号融合**：麦克风检测"整齐回答""窃窃私语"补充姿态维度；
- **注意力热力图**：把全班专注度投影到座位布局俯视图；
- **教师动作识别**：把板书/提问/走动等教师行为也纳入分析，做课堂互动性的归因；
- **多教室对比**：把多个教室的课堂数据横向对比，帮助教研组发现普遍性问题。

### Q20. 如果让你现在从零重做，会怎么改？

**A.** 技术上会做两件事：
1. 把行为判定从规则引擎 + CNN 做成"**规则打底 + 轻量 MLP 修正**"的混合体——规则负责可解释的大类判别，MLP 负责边界 case 的细化；
2. 把 AI 推理拆到独立进程（而不是线程），用 ZMQ / shared memory 做 IPC，完全绕过 Python GIL；对多摄像头场景能进一步提吞吐。

## 五、常见"追问"应对

> 评委可能在 Demo 中即兴追问，此处列典型问题与简短回答。

- **"那个框为什么跳？"** → 检测 FPS 和预览 FPS 不一样，这是架构选择，能让你看到摄像头原生流畅度。
- **"我现在趴桌了但它显示专注！"** → 窗口内多数票机制，要持续 2-3 帧才切换，避免误识别。
- **"关灯会崩吗？"** → 不会，只是检测不到人，分数归零，画面仍实时推。
- **"能跑树莓派吗？"** → yolov8n-pose + imgsz=416 可以，估计 2-3 FPS，能做演示。
- **"数据能导出吗？"** → 数据库是标准 SQLite，任何工具都能打开；也可以加一个 CSV 导出接口。
- **"LLM 没配置会怎样？"** → 仪表盘右侧助教卡片会显示"未配置"，其它功能正常。
